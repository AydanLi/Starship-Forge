# 星舰熔炉 · 抖音上线前改进方案(v2)

> 定位:承接《工程改进路线图.md》的评审结论,吸收方案讨论后的修订(跨波保留改为**刚体回流**、构筑可控改为**软权重+三选一**、Boss 机制**数据化**),重排为「上线抖音前必须完成」的执行文档。每个工作包给出规则、实现、验收、风险;文件映射沿用路线图第 2 节。

## 0. 上线判定(Definition of Done)

满足以下全部条件才提审:

1. 核心循环规则自洽:跨波舰队规则落地,新玩家 3 局内能理解「合成→编队→战斗→回炉」的资产循环。
2. 构筑有方向感:玩家能主动追一套羁绊,广告点位挂在「关键一张」上。
3. 五个 Boss 各有 1 个可见机制,且都与战术技时机相关。
4. 随机可复盘:固定 seed 结果稳定,核心规则测试可一键运行。
5. 真机闭环:tt.login、真实广告位、埋点、分享战报卡、音频分包全部通过真机验证。
6. 合规材料齐备:软著(或电子版权认证)、隐私政策、内容自审表。

## 2. WP1 跨波舰队 —— 刚体回流方案(P0)

### 2.1 为什么不用「token 回流待编区」

熔炉是本作核心乐趣。token 方案下中后期玩家只需维持 6 格阵容,熔炉退化为补员工具;且 `board.enter()` 每次重建 slots/bench,是路线图承认的最高风险点。改为刚体回流后:

- 跨波资产与熔炉玩法统一:上一波战舰留在炉内,可继续与同级碰撞合成升级,「合成链」与「升星」两套成长自然衔接;
- `board.enter()` / `deployables()` 完全不动,最高风险点消失;
- 过载线从纯惩罚变成**舰队规模上限的物理表达**:舰队越攒炉子越满,玩家必须主动合成精简——策略资源管理是涌现出来的,零额外代码。

### 2.2 产品规则

- 胜利结算时,存活的非召唤上阵单位**变回带标签刚体**,从投放口依次掉回熔炉(有掉落动画,题材=舰队返航入坞)。
- 星级保留:刚体新增 `gStar` 字段;`unitFromToken` 时星级来自刚体。
- 死亡不回流;召唤物不回流;失败重试本波不结算损耗(回到本波备战,资产以开战前存档为准)。
- **残骸回收**:阵亡战舰按 `C.VALUE[tier] × 星级系数 × 回收率(建议 0.3)` 折算金币,随胜利结算发放。输出「损失 N 艘,回收残骸 +M💰」,缓解「死亡=永久损失」的挫败,也给经济一个回路。
- 物理合成与星级:同级刚体碰撞合成时,**忽略星级、取两者较高星**(简单、可解释);同级同星的精确升星仍走编队拖拽。

### 2.3 实现步骤

1. `forge.addBall(tier, x, y, vy?, fac?, cls?, star?)` 增加可选标签参数(招募援军已在用该入口,向后兼容)。
2. `battle.start()` 为每个上阵单位记 `u.srcTier/srcFac/srcCls/srcStar`(只记值,不引用 token,避免幽灵字段)。
3. `battle.end('win')`:
   - survivors = 存活 && !summon && 有 src 记录;
   - 逐艘 `forge.addBall(srcTier, 随机x, Y_DROP, 0.5, srcFac, srcCls, srcStar)`(间隔 0.15s 依次掉落,视觉上是返航);
   - 阵亡者累计残骸金币,并入 `G.lastGain` 单独显示。
4. `save`:存档增加熔炉刚体快照(tier/fac/cls/star/x),版本号 +1,旧档迁移为空炉。
5. 难度重调:普通敌 HP/ATK 按星区补 10%~25% 斜率,Boss 波再上浮;数值全部进 `config.ts`(新增 `ENEMY_SCALE`),不散落在 `fleet.genEnemies()` 里。

### 2.4 验收标准

- 第 1 波胜利后,存活战舰(含星级)出现在第 2 波熔炉中,可直接编队,也可与同级合成;
- 死亡/召唤单位不回流;阵亡回收金币数额正确且有 UI 提示;
- 失败重试回到本波开战前状态,无重复/幽灵单位;
- 回流刚体堆到过载线同样触发过载(风险自担的舰队规模上限);
- 清档/返回主界面/续档三条路径下熔炉快照一致。

### 2.5 风险

- 雪球:回收率与敌方 scale 是两个平衡阀门,先保守(回收 0.3、scale 偏高),用 WP4 的 seed 测试跑数值曲线再松;
- 存档体积:刚体快照按上限 ~30 艘估算,远小于 1KB,无压力;
- 高级刚体(母舰 r=118)回流多艘可能瞬间过载:回流前检查堆高,超限部分自动转为残骸回收(提示「船坞已满,N 艘折价回收」)。

## 3. WP2 构筑可控 —— 软权重 + 三选一(P0)

### 3.1 软权重(先做,半天)

`forge.addBall()` 生成 fac/cls 时改为伪随机加权:统计当前可上阵资产(熔炉 5 级以上刚体)中各阵营/舰种数量,已拥有 ≥2 个的类别权重 ×`SOFT_W`(建议 2.0,进 config)。TFT 商店同款思路,只改一个随机函数,「想凑 4 帝国」从纯运气变成可经营。依赖 WP4 的 rng 注入以便测试分布。

### 3.2 三选一定向招募(替换现有「招募援军」)

- 点「招募援军 15💰」→ 弹 3 张候选卡,各显示**阵营/舰种/等级**(等级 = DEPLOY_MIN 或 +1);
- 候选生成规则:至少 1 张命中「玩家当前最缺的羁绊缺口」(距离 2/4 档差 1 时优先);
- **广告点位重排**:原「📺 免费援军」升级为「📺 免费定向招募」——看广告获得一次三选一。这是全场动机最强的广告位(就差一张时的转化率远高于白给一艘无人机);
- 金币不足/广告失败:明确反馈,不发奖励,不卡流程。

### 3.3 涉及模块与验收

`economy.ts`(候选生成+入口)、`state.ts`(`G.panel='recruit'` + 候选列表)、`GameApp.ts`(候选弹窗绘制与命中)、`config.ts`(SOFT_W/候选规则)、`ads.ts`(复用)。

验收:选择候选后顶部羁绊条实时更新;软权重开关(SOFT_W=1)可退化为纯随机;固定 seed 下候选序列可复现;广告只发一次奖励。

## 4. WP3 Boss 机制数据化(P1)

### 4.1 原则

不写 5 个 Boss 专属函数;`config.ts` 声明式描述,`battle.ts` 实现 5 个通用 handler。**每个机制都是「战术技时机」的考题**——战斗中玩家唯一的主动操作就是那个按钮,全部深度都往这一个决策点上压。

```ts
// config.ts
BOSS_SKILLS: [
  { type: 'summon',  threshold: 0.5, count: 2 },              // S1 秃鹫号:半血召唤护卫 → 留技能清场
  { type: 'shield',  interval: 8, value: 800 },               // S2 铁卫:周期护盾 → 破盾窗口再按
  { type: 'hatch',   interval: 6, count: 2 },                 // S3 深渊之母:持续孵化 → 攒队清潮
  { type: 'silence', interval: 10, duration: 4 },             // S4 零号意志:锁战术技 → 逼你提前按
  { type: 'snipe',   interval: 7, mult: 3.0 },                // S5 万王之王:点名前排 → 抢在斩杀前爆发
]
```

### 4.2 实现与验收

`battle.update()` 调 `updateBossSkill(boss, dt)`,按 `BOSS_SKILLS[G.level].type` 分发;每次施放必须有三件套:飘字/特效(fx)+ 音效(audio,可复用 tactical/overload)+ 施放前 1s 的预警提示(如 Boss 外圈闪烁)。沉默机制需在 `uiModel()` 中把战术技按钮文案改为「⚠ 被零号意志压制 Ns」。

验收:每个 Boss 机制真机可见、有预警、不卡死;25 秒超时判定不受影响;S4 沉默期间按钮态正确;机制数值全部可在 config 一处调整。

## 5. WP4 RNG / 测试基建(P1,建议最先做)

### 5.1 core/rng.ts

```ts
export interface Rng { next(): number; int(max: number): number; pick<T>(a: T[]): T; range(a: number, b: number): number; }
export function makeRng(seed?: number): Rng   // mulberry32;无 seed 时用 platform.now() 播种并记录
```

逐步替换:`pickDropTier` → `forge.addBall` 标签 → `battle.pickTarget`/暴击/溅射。**seed 写入存档与战报**。

### 5.2 测试(vitest,仓库根,零 Cocos 依赖)

core/ 零 cc 依赖就是为此准备的。用例清单:羁绊 2/4 档 ×8、召唤不计羁绊、同级同星升星与 3 星封顶、战术技公式(×2.4 均分)、胜利金币(普通/Boss 波)、广告奖励三点位、**WP1 全部验收项(回流/不回流/残骸回收/重试一致性)**、软权重分布(固定 seed 卡方检验式断言)。CI 可后补,先保证 `npx vitest` 本地一键绿。

### 5.3 顺手的传播功能(成本极低,上线加分)

- **每日挑战**:每天全网同一 seed(按日期哈希),同一投放序列 → 天然可比分数与话题;入口放主菜单,可作为上线后的第一个运营抓手;
- **战报卡**:波次/羁绊/最高星/击杀数据都在,Painter 画离屏 RT 导出图 → 喂 `tt.shareAppMessage`(WP5)。

## 6. WP5 抖音真机闭环(P0,与玩法并行)

### 6.1 技术清单(建议顺序)

1. **音频分包**(影响现有代码):把 `resources/audio/` 移出 resources,建独立 Asset Bundle `audio` 并在构建设置里勾为子包;`view/Bgm.ts` 从 `resources.load('audio/…')` 改为 `assetManager.loadBundle('audio')` 后再 `bundle.load(...)`,加载失败静默(保持现有回退策略)。主包目标 <4MB,tex 5MB 也需要评估进分包或压缩(webp/更高压缩率)。
2. **登录**:`user.ts` 接 `tt.login` → code 换 openid 作 uid(无后端阶段可直接用 `anonymousCode`/code 落地为稳定 uid),存档 key 结构不变;保留代号登录作为浏览器预览通道(platform 分流)。
3. **广告**:`platform.ts` 填真实激励视频 adUnitId;补齐加载失败兜底(onError → done(false) 已有,补「广告暂不可用」提示与按钮置灰);同一点位加频控(config)。
4. **埋点**:新增 `core/analytics.ts`,浏览器 console.mock / 真机 `tt.reportAnalytics` 双实现走 platform 分流。事件表:`login_start/success`、`run_start`、`wave_start/win/lose(带 level,wave,seed)`、`boss_start/win/lose`、`ad_show/complete/fail(带点位)`、`share_show/success`、`recruit_pick(带候选与选择)`。埋点从第一天就带 seed,方便回查战报。
5. **分享战报卡**:Boss 胜利与通关结局后出「分享战报」按钮 → 离屏渲染卡图 → `tt.shareAppMessage`;分享成功事件入埋点。奖励挂钩(分享得金币)先不做,规避审核风险,上线后看数据再说。
6. 真机回归:走通「登录→序章→S1 三波→广告三点位→分享→杀进程续档」全链路。

### 6.2 合规材料(第一天启动,周期最长)

- 软著或电子版权认证(提审硬门槛,办理 2~4 周,**立即启动**);
- 游戏内容自审自查表、隐私政策文本(登录/存储/广告 SDK 声明);
- 广告场景截图与说明(三个点位均为用户主动触发,符合激励视频规范);
- 详见 `docs/抖音小游戏发行盈利指南.md`。

### 6.3 验收标准

真机稳定拿到 uid;断网/无广告填充不卡流程;主包 <4MB、总包 <20MB;每个广告点位曝光与完成事件对得上;Boss 胜利可分享出卡图;冷启动续档正确。

## 8. 工程纪律(沿用路线图第 10 节并追加)

- 新增数值一律进 `config.ts`;平台能力一律走 `platform.ts`;Boss 机制/敌方 scale/回收率/软权重全部数据化。
- 先改 Cocos 主力版,web 原型仅在验证玩法手感时同步(WP1 建议同步,便于快速试数值)。
- `battle.end()` 是回流与埋点的统一入口,不要把结算逻辑散进 flow。
- Bgm/Tex 的「资源缺失静默回退」原则同样适用于新增的候选弹窗、战报卡与分包加载。
