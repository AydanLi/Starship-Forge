# 《星舰熔炉》工程改进路线图

> 目的：把当前设计评审转成工程团队可执行的开发文档。本文面向 Cocos 主力版本，同时兼顾 Web 原型的逻辑参照。

## 1. 当前结论

项目已经具备可玩的核心闭环：

- 熔炉投放与同级合成。
- 5 级以上战舰生成阵营与舰种标签。
- 6 格编队、待编区、拖拽布阵、同级同星升星。
- 4 阵营与 4 舰种羁绊。
- 自动战斗、前排优先承伤、战术技、胜负结算。
- 固定金币经济与 3 个激励视频点位。
- Cocos 版正式美术、背景音乐、登录、存档、新手引导、星图。

当前主要问题不是缺少更多内容，而是以下 5 件事还没有完全产品化：

1. 跨波舰队规则不清晰，当前更像“本波消耗制”，不完全符合自走棋的阵容经营预期。
2. 阵容构筑可控性不足，玩家很难主动追求某个阵营或舰种。
3. Boss 主要是数值墙，星区差异和记忆点不足。
4. 战斗随机不可复盘，自动化测试资产未入库或不可见。
5. 抖音上线闭环缺少登录、广告位、埋点、分享战报卡和资源分包确认。

## 3. P0：先修正跨波舰队规则

### 背景

GDD v2 描述的是“阵亡的战舰保留、下一波继续强化”的自走棋节奏，但当前开战时会把上阵 token 对应的熔炉刚体移除，战斗结束后 `slots` 与 `bench` 清空，下一波重新从熔炉开始。这会削弱长期 build 感，也会让“升星”和“羁绊经营”的价值只停留在单波。

### 推荐产品规则

建议改成“存活舰队跨波保留”：

- 开战消耗熔炉中的刚体，但战斗单位在胜利后回流为 token。
- 存活单位进入下一波待编区或自动回到原阵位。
- 已死亡单位不回流，召唤物不回流。
- 星级、等级、阵营、舰种保留。
- 当前波未上阵的 `bench` 单位保留。
- 失败重试本波时，不保留本次失败战斗中的损耗，回到本波重新备战。

这比“所有战舰永久保留”更容易形成损耗与补强，也不会让早期阵容雪球失控。

### 实现建议

在 `battle.start()` 进入战斗前，为每个上阵单位记录来源 token：

```ts
u.sourceToken = G.slots[i];
u.sourceSlot = i;
```

在 `battle.end('win')` 时生成下一波可用舰队：

```ts
const survivors = G.pUnits
  .filter(u => u.alive && !u.summon && u.sourceToken)
  .map(u => ({
    ...u.sourceToken,
    bodies: [],
    x: 0,
    y: 0,
  }));
```

然后由 `flow.nextAfterWin()` 或新增的 fleet carryover 方法在进入 `PREP` 后恢复：

- 简单版：全部 survivor 放入 `G.bench`。
- 体验版：按 `sourceSlot` 尝试回到原阵位，进入 `DEPLOY` 时可见。
- 推荐 MVP：胜利后回到 `PREP`，保留单位显示在待编区，玩家仍需点“编队部署”调整。

需要注意：当前 `board.enter()` 会从 `forge.deployables()` 重新生成 token，并重置 `G.slots/G.bench`。如果要跨波保留，需要让 `board.enter()` 合并已有 `G.bench` 与熔炉新生成战舰，而不是直接清空。

### 验收标准

- 第 1 波胜利后，存活的非召唤战舰在第 2 波仍可上阵。
- 升星后的战舰跨波保留星级。
- 已死亡战舰不回流。
- 召唤单位不进入下一波。
- 未上阵待编单位不丢失。
- 失败重试不会把失败战斗前后的状态混在一起。

### 风险

- 如果保留过强，后期难度会被滚雪球压平。
- 需要重新调普通敌与 Boss 血量。
- `board.enter()` 的清空逻辑要小心改，避免重复 token 或幽灵单位。

## 4. P0：补一层构筑可控性

### 背景

当前可上阵战舰的阵营与舰种在 `forge.addBall()` 中随机生成。玩家可以通过刷新投放队列影响等级，但很难影响标签。结果是“想凑 4 帝国或 4 炮舰”更多靠运气，广告和金币消费也缺少“就差一张”的强动机。

### 推荐方案

先做轻量“战术改造”系统，不做复杂商店：

- 新增按钮：`战术改造 10 金币`。
- 对选中的待编或阵位战舰，弹出 3 个选项：
  - 改阵营为随机 1 个未拥有阵营。
  - 改舰种为随机 1 个未拥有舰种。
  - 升级一次标签品质或获得临时攻击加成。
- 免费广告点位可给一次战术改造。

如果想更省实现成本，可以先做“定向招募”：

- 原“招募援军”改为三选一。
- 每个选项显示阵营、舰种、等级。
- 付费招募与广告招募都走同一套候选生成。

### 涉及模块

- `economy.ts`：新增消费入口与广告入口。
- `board.ts`：支持选中 token。
- `state.ts`：增加当前选中 token 或候选列表。
- `GameApp.ts`：绘制候选弹窗与点击区域。
- `config.ts`：新增改造费用、候选数量、可选权重。

### 验收标准

- 玩家能明确选择某个方向补羁绊。
- 改造或招募后，顶部羁绊提示实时更新。
- 金币不足时有明确反馈。
- 广告完成后只发放一次奖励，关闭或失败不发奖励。

## 5. P1：让 5 个 Boss 具备差异机制

### 背景

当前 Boss 主要由 `fleet.genEnemies()` 生成，差异集中在 HP、攻击和护卫。这样前几局可玩，但录屏传播和长期记忆点不足。

### 推荐 Boss 机制

| 星区 | Boss | 机制 | 工程实现 |
| --- | --- | --- | --- |
| 1 残骸带 | 秃鹫号 | 低血量时召唤 2 个护卫 | `battle.update()` 中检测 HP 阈值，调用召唤 |
| 2 机械矩阵 | 铁卫 | 每 8 秒获得护盾 | 给 Boss 加 `skillTimer` 与 `shield` |
| 3 虫巢星云 | 深渊之母 | 周期孵化虫群小怪 | 定时向 `G.eUnits` push 小怪并 layout |
| 4 深渊 AI | 零号意志 | 延长玩家战术技冷却或短暂沉默 | 修改 `G.tacticalCd` 或加 `tacticalLocked` |
| 5 帝国终末 | 万王之王 | 点名前排造成高额伤害 | 定时选择我方 front 单位 hit |

### 实现建议

新增 Boss 技能分发函数，避免把所有逻辑堆进 `battle.update()`：

```ts
function updateBossSkills(dt: number): void {
  const boss = G.eUnits.find(u => u.alive && u.isBoss);
  if (!boss) return;
  switch (G.level) {
    case 0: updateVulture(boss, dt); break;
    case 1: updateIronGuard(boss, dt); break;
    case 2: updateHiveQueen(boss, dt); break;
    case 3: updateZeroWill(boss, dt); break;
    case 4: updateStarKing(boss, dt); break;
  }
}
```

### 验收标准

- 每个 Boss 至少有 1 个可见机制。
- 机制有 UI 或特效反馈，不只是数值变化。
- Boss 技能不会导致战斗卡死。
- 25 秒超时规则仍可正常判胜。

## 6. P1：补可复盘随机与自动化测试

### 背景

当前投放、标签、暴击、寻敌、溅射目标等都直接用 `Math.random()`。这会导致数值调试难复现，也不利于回归测试。移植说明提到“12 项无头回归测试”，但仓库里没有可见的 test/spec 文件。

### 推荐方案

新增 `core/rng.ts`：

```ts
export interface Rng {
  next(): number;
  int(max: number): number;
  pick<T>(items: T[]): T;
}
```

默认用 `Math.random()`，测试时可注入 seed。逐步替换：

- `config.pickDropTier()`
- `forge.addBall()` 的阵营/舰种随机。
- `battle.pickTarget()`。
- 暴击与溅射目标。

### 测试清单

优先补以下用例：

- 羁绊计数：召唤物不计入羁绊。
- 帝国、异星、机械、赛博 2/4 档效果。
- 突击、炮舰、无人机、辅助效果。
- 2 个同级同星 token 升星，3 星封顶。
- 战术技伤害公式：全队攻击乘 2.4 再按敌数均分。
- 胜利金币：普通波与 Boss 波公式。
- 广告奖励：金币双倍、旗舰超载、免费援军。
- 跨波保留：存活单位回流，死亡单位不回流。

### 验收标准

- 本地能一键运行测试。
- 测试不依赖 Cocos 编辑器。
- 随机相关测试固定 seed 后结果稳定。

## 7. P1：补抖音上线闭环

### 当前缺口

- `platform.ts` 中激励视频广告位还是 `REPLACE_WITH_YOUR_AD_UNIT_ID`。
- 登录还是代号或游客，未接 `tt.login` 与 openid。
- PRD 要求埋点，但当前未见完整 analytics 模块。
- M4 中的分享战报卡和 `tt.shareAppMessage` 仍未实现。
- 音频资源约 8.5MB，需要确认分包，否则首包风险较高。

### 建议实现顺序

1. 新增 `analytics.ts`，先本地 console/mock，后续替换平台上报。
2. 接入关键事件：
   - `login_start`, `login_success`
   - `run_start`
   - `wave_start`, `wave_win`, `wave_lose`
   - `boss_start`, `boss_win`, `boss_lose`
   - `ad_show`, `ad_complete`, `ad_fail`
   - `share_card_show`, `share_success`
3. 替换 `user.ts` 登录为 `tt.login` 到 openid。
4. 填广告位 ID，并处理广告加载失败兜底。
5. 做通关或 Boss 胜利战报卡。
6. 检查 Cocos 构建分包配置，优先把 `resources/audio/` 放分包。

### 验收标准

- 真机能登录并拿到稳定 uid。
- 广告失败不会卡流程。
- 每个广告点位都有曝光与完成事件。
- Boss 胜利后可触发分享。
- 首包体积满足抖音限制。

## 8. P2：长期内容扩展

以下内容建议等 P0/P1 跑通后再做：

- 图鉴、成就、每日任务。
- 更多战术技。
- 阵营克制。
- 赛季规则或高难模式。
- 皮肤涂装。
- 排行榜与好友阵容展示。

这些功能会增加留存深度，但在核心循环和数据闭环未稳定前，不应优先投入。

## 9. 推荐里程碑

### M1：规则对齐

- 完成跨波舰队保留。
- 梳理失败重试、返回主界面、清档后状态。
- 补对应测试。

### M2：构筑可控

- 完成定向招募或战术改造。
- 调整金币产出和消费。
- 验证“差一件凑羁绊”的玩家动机。

### M3：Boss 记忆点

- 每个星区加入 1 个 Boss 机制。
- 给机制加文本、特效或音效提示。
- 调整 5 星区难度曲线。

### M4：抖音闭环

- 接 `tt.login`、真实广告位、埋点、分享战报卡。
- 处理音频分包。
- 真机跑完整流程。

### M5：小规模数据验证

- 看关卡漏斗、广告完成率、分享触发率。
- 只基于数据决定是否扩内容。

## 10. 工程注意事项

- Web 原型与 Cocos 版当前逻辑一一对应。修改玩法规则时，建议先改 Cocos 主力版本，再决定是否同步 Web 原型。
- `board.enter()` 当前会重建 slots 和 bench，跨波保留时这是最高风险点。
- `battle.end()` 当前只处理胜负结果和金币，适合作为跨波回流的入口。
- 所有新增经济数值先放入 `config.ts`，不要散落在 UI 或流程代码。
- 所有平台能力先走 `platform.ts` 或新增平台适配模块，不要在玩法模块里直接调用 `tt.*`。
- Boss 技能应尽量数据化，避免后续每加一个 Boss 都改大段战斗循环。
