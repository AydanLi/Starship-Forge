# 星舰熔炉 · 架构审视报告(M3 后全量复查)

> 范围:Cocos 主力版全部 25 个 TS 文件(~2300 行)+ 资产管线 + 测试基建;从分层、状态管理、确定性、平台边界、性能、扩展性六个维度逐文件审读。web 原型(js/)按「历史参照」对待,见 §5。
> 结论先行:**分层架构本身健康,无需推倒的硬伤;审出 3 个帧率/时钟类真 bug(已随本报告修复),其余为分级列出的结构性债务与扩展建议。**

## 1. 架构现状与优点(值得保持的决策)

```
数据层   config.ts / storyData.ts          纯数据,零依赖
服务层   rng / platform / save / user / audio / analytics(未来)
系统层   state(G) ← forge / board / battle / synergy / fleet / economy / ads / story / tutorial / bgm
装配层   flow.ts                            状态机 + 输入路由 + 系统接线(bind 回调,无反向依赖)
视图层   view/  GameApp(唯一组件) + Painter(立即模式画笔) + Tex/Bgm(资源库)
```

- **core 零 cc 依赖**是全项目最有价值的决策:49 项 vitest 无头回归直接跑 node,物理用 20 行桩;每个里程碑的规则改动都有安全网。
- **单向依赖**:core 不知道 view;view 只读 G/调 core 方法;跨系统回调经 flow 的 bind 注入,无环(save→forge 为后加的单向,合法)。
- **数值全量数据化**:TIERS/ENEMY/SALVAGE/SOFT_W/BOSS_SKILLS(将至)全在 config,平衡迭代不碰逻辑。
- **可播种 rng + seed 入档**:复盘、测试、每日挑战三件事共用一个地基。
- **资源缺失静默回退**(Tex 缺图回矢量、Bgm 缺曲静音):加载失败永不卡游戏,也让分包/裁剪变成纯配置问题。
- **命中区数据放 config**(BTN_DOUBLE/RC.CARDS):core 可测试点击语义,view 只描画,这个模式请在后续面板沿用。
- **资产管线可复现**:art_raw 原图 → 边缘泛洪键控 → pngquant,坏一张随时单独重出。

## 2. 硬伤(P0,已随本报告修复 ✅)

| # | 问题 | 危害 | 修复 |
|---|---|---|---|
| 1 | `forge.update` 每帧固定步进 `Engine.update(1000/60)` | 物理速度=帧率:90/120Hz 真机上熔炉 1.5~2 倍速,合成手感/过载判定全变 | 固定步长累加器(1/60),dt 上限 0.25s 防死亡螺旋 |
| 2 | `forge.drop` 用 `setTimeout(420ms)` 重生下一件 | 逃逸游戏循环:广告/剧情打开时照跑;确定性回放不可能;极端时序下跨阶段触发 | 改为循环内 `respawnT` 计时,随 flow.step 的门控自然暂停 |
| 3 | core 内直接用 `performance.now()`(刚体 born/过载计时) | 平台 API 潜入 core,违反「只有 platform.ts 感知环境」;回放不可确定 | 改为模块内模拟时钟 `simTime`(随 update 累加) |
| 3b | `fx.update` 粒子位移不乘 dt | 高刷屏粒子飞 2 倍速(纯视觉) | 按 dt×60 缩放,60fps 手感不变 |
| 3c | Painter 每帧数百次 `new Color`(parseColor/img) | 低端机 GC 抖动 | 颜色缓存(Map,512 上限);无功能变化 |

以上修复已通过 49 项回归 + tsc。**battle.update 本来就是 dt 驱动的,不受影响。**

## 3. 结构性风险(P1:不修不炸,但决定扩展成本)

**3.1 `G: any` 全局可变状态是最大技术债。** 所有模块自由读写无类型的 G,字段契约靠注释。眼下 55 行还能背下来,加图鉴/成就/每日挑战后会失控。建议(低成本高收益,~半天):建 `core/types.ts` 定义 `ShipTag{tier,fac,cls,star}`、`Unit`、`Token`、`Offer`、`GState`,把 `G: any` 换成 `G: GState`;**不改任何运行逻辑**,只是把隐式契约显式化。附带收益:立刻暴露 3.2。

**3.2 「舰船四元组」有两套字段名。** `{tier,fac,cls,star}` 出现在 token/src/offer/unit 里,而存档快照用缩写 `{t,f,c,s,x}`——同一概念两套拼写,是未来 bug 温床(已在 forge.snapshot/restore 两端小心对齐,但第三个使用者出现时必错)。建议:types.ts 里配一对 `packShip/unpackShip` 编解码函数,快照格式不变(存档兼容),代码里只流通全名。

**3.3 flow.ts 的 switch 正在长毛。** uiModel/onFire/pointerDown 三处 per-phase 分支,M3 已经开始塞面板特判(`G.panel==='recruit'`),M4 的沉默按钮态、未来的图鉴面板还会继续塞。阈值建议:**再加一个面板就重构**为 phase→handler 表(`PHASES[phase] = {fire, reset, down, ui}`),面板走统一的 panel 栈(open/close/input 一套约定),switch 变查表。现在不动,它还没痛。

**3.4 battle.end 职责在膨胀。** 已承担:回流+残骸+船坞上限+金币+存档+提示;M4 埋点、战报数据还要进来。建议 M4 动 battle 时顺手把胜利结算抽成 `settleWin(): {returned,lost,salvage}` 纯函数,end 只做编排——测试也能从「摆 units 调 end」简化为直接断言 settle 结果。

**3.5 模块级单例限制「多实例模拟」。** forge 的 engine/world、ads 的 adSim、bgm 的播放态都是模块闭包单例。单局游戏没问题(这也是它简单的原因),但意味着:同进程跑不了两个对局(每日挑战 ghost 回放、后台数值模拟器、A/B 平衡跑批)。不建议现在改;若未来要数值模拟器,方案是把 battle+synergy+fleet(本就近乎纯函数)抽成可实例化的 sim 包,forge 物理不进模拟器(用快照近似)。

**3.6 单条随机流。** 掉落、标签、招募、战斗共用一条 rng 流:玩家任何一次「刷新队列」都会使同 seed 的两局从此分叉。对复盘无碍(回放含操作序列),但对**每日挑战的公平性**有细微影响(操作不同→后续掉落不同,这其实是 roguelike 常态,可接受)。若想要「当日掉落序列全网严格一致」,届时把 rng 拆两条流(drop 流/combat 流)即可,rng.ts 已预留 makeRng 思路。

## 4. 一致性债务(P2:记录在案,顺手再还)

- **audio.ts 在 core 里直接摸 WebAudio**(globalThis.AudioContext),违反「只有 platform 感知环境」。现状可用(有存在性守卫,node/真机自动静音),M5 换真实音效文件接 AudioSource 时自然消除;在此之前视为**已豁免的例外**,不要新增第二个。
- **UI 几何混在 core**(menu.ts 的 MENU_UI、config 的 SLOTC/RC/BTN_*)。这是「命中区共享」模式的代价,可接受;若做多分辨率/横屏再统一收拢成 C.UI 命名空间。
- **存档单键覆盖写**,无备份槽:真机写入途中被杀进程理论上可丢档。M5 接 tt 存储时加双缓冲(key_A/key_B+代数)即可,~20 行。
- **GameApp 579 行**:尚可维护;超过 ~800 行或新增第 4 个全屏面板时,按 drawXxx 拆 `view/screens/`。
- **Painter.text 每帧重设 Label 全属性**(cacheMode NONE):当前文字量(~40 个/帧)没问题;若未来帧率吃紧,金币/分数等高频数字改 CacheMode.CHAR。
- **GameApp.update 的 `_err` 只报一次错然后每帧静默吞**:真机排障时容易黑屏无线索。建议 M5 埋点时把 render error 也上报。

## 5. web 原型(js/)已经失真

README 声称「两版玩法逻辑一一对应」,但 M1~M3(rng/回流/残骸/软权重/三选一)都只进了 Cocos 版。**这是文档硬伤**(会误导以后翻 js/ 对照的人)。两个选项:(a) 花 ~1 天把 M1~M3 反向移植回 js/;(b) README 改口为「web 原型 = M0 玩法参照,新规则以 Cocos 版为准」,原型降级为快速试玩壳。建议 (b)——上线前的工时应该花在 M4/M5,不是维护双端。**本报告随附的 README 修改采用了 (b),如果你想要 (a) 告诉我。**
