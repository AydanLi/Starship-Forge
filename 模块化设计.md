# 《星舰熔炉》模块化设计方案

> 状态：**已实施**（按「七、分步重构计划」完成拆分，`game.js` 已删除）。本方案记录：把当年 ~1100 行的单体 `game.js` 拆成职责单一、可独立修改的模块，同时**不引入任何构建工具**、**不破坏双击即玩**、**为 Cocos TypeScript 迁移铺路**。现 Web 原型 `js/*.js` 已按此结构落地，Cocos TS 移植（`cocos/assets/scripts/`）亦在其上展开。

## 一、原始现状问题（拆分前）

拆分前，`game.js` 是一个 IIFE 单体，混着 12 类职责：数值配置、剧情数据、熔炉物理、编队拖拽、羁绊计算、自动战斗、经济、广告、特效、渲染（约占 40%）、输入路由、主循环。当年的实际痛点：改一处数值要在千行里找；渲染和逻辑纠缠，之后换美术要大动；新功能（音效/存档/新星区）只能继续往里塞；多人协作或外包接手成本高。上述问题已通过本方案的拆分解决。

## 二、方案选型：多脚本 + 命名空间（不用 import/export）

三个候选里选了最朴素的一个，理由如下：

| 方案 | 双击 index.html | 需要构建工具 | 单文件版构建 | Cocos 迁移 |
|------|----------------|--------------|--------------|-----------|
| **多 script 标签 + SF 命名空间（选定）** | ✅ 可玩 | ❌ 不需要 | 按顺序拼接，脚本改 3 行 | 每个模块 → 一个 TS 类/服务 |
| ES Modules (import/export) | ❌ file:// 被 CORS 拦 | 开发不用、打包要 esbuild | 需引入打包器 | 最顺，但代价在前两列 |
| 打包器工程 (Vite/esbuild) | ❌ | ✅ | ✅ | 顺 |

原型阶段「零工具链 + 到处能跑」价值最大；等正式迁 Cocos 时本来就要换成 TS 工程，现在没必要提前引入打包器。

**命名空间约定**：全局只挂一个 `window.SF`（StarForge）。每个文件是一个 IIFE，只往 `SF.<模块名>` 上挂自己的公开接口，文件内部实现全部私有。禁止模块 A 直接写模块 B 的内部变量——跨模块只调公开函数。

## 三、目录结构与模块职责

```
D:\Game project\
├─ index.html          （按依赖顺序引入 js/*.js）
├─ js/
│  ├─ config.js        数据层：全部数值配置（无依赖）
│  ├─ story-data.js    数据层：剧情文案（无依赖）
│  ├─ state.js         数据层：运行时状态 G + phase 切换
│  ├─ user.js          数据层：指挥官代号/游客身份（uid）
│  ├─ save.js          服务层：按 uid 分账号存档读写

│  ├─ fx.js            服务层：粒子/光束/飘字/屏震 特效池
│  ├─ synergy.js       服务层：羁绊计算（纯函数）
│  ├─ fleet.js         服务层：单位工厂（token/unit/召唤/敌军）
│  ├─ forge.js         系统层：熔炉（matter 物理/投放/合成/过载）
│  ├─ board.js         系统层：编队（阵位/拖拽/升星）
│  ├─ battle.js        系统层：自动战斗（寻敌/结算/波次/战术技）
│  ├─ economy.js       系统层：金币/刷新/招募/结算奖励
│  ├─ ads.js           系统层：广告点位（模拟↔真SDK 唯一替换点）

│  ├─ menu.js          系统层：登录/主界面/星图/设置

│  └─ main.js          装配层：startRun / 主循环 / 模块接线

```

## 四、依赖分层（只允许上层调下层，杜绝循环依赖）

```
装配层   main.js ──────────── 接线、主循环、启动
交互层   input.js  ui.js ──── 把点击翻译成系统层调用
表现层   render.js ─────────── 只读 G 和各系统的查询接口，纯输出
系统层   forge  board  battle  economy  ads  story ── 玩法规则，可互相通过 G 协作
服务层   fx  synergy  fleet ── 无状态/自包含，被系统层调用
数据层   config  story-data  state ── 被所有层读取，不依赖任何人
```

两条铁律：**render 不改状态**（只画）；**数据层不 require 任何模块**。系统层之间尽量通过 `SF.G.phase` 和 main 的接线协作，避免互相深调。

## 六、迁移 Cocos TypeScript 的映射

实际 Cocos TS 移植落在 `cocos/assets/scripts/`：`core/*.ts`（逻辑）与 `view/*.ts`（绘制）。映射如下（对照现有 `.ts` 文件）：

| 原型模块 | Cocos 落点（实际 `.ts`） |
|----------|-----------|
| config.js / story-data.js | `core/config.ts` / `core/storyData.ts`（TS 常量，对接数值表） |
| state.js | `core/state.ts`（GameState / 运行时状态 G） |
| user.js | `core/user.ts`（指挥官代号/游客 uid） |
| save.js | `core/save.ts`（按 uid 分账号存档，带校验） |
| audio.js | `core/audio.ts`（音效/BGM 服务） |
| forge.js | `core/forge.ts`（ForgeSystem；matter → 内置 Box2D，刚体/碰撞回调概念一致） |
| board.js / battle.js | `core/board.ts` / `core/battle.ts` |
| synergy.js / fleet.js | `core/synergy.ts` / `core/fleet.ts`，纯 TS 工具，**原样复制即可用** |
| economy.js / story.js | `core/economy.ts` / `core/story.ts` |
| ads.js | `core/ads.ts`：激励视频封装，走 `platform.showRewardedAd`（对接点收敛在此，正式版只换它） |
| menu.js | `core/menu.ts`（登录/主界面/星图/设置） |
| （新增）flow.ts | `core/flow.ts`：对局流程编排（launch/开战/结算/返回主界面 toMenu，即原 main.js 的接线职责） |
| （新增）platform.ts | `core/platform.ts`：三端适配层，抽象**存储 / 激励视频 / 键盘**（浏览器 ↔ 抖音 `tt.*`） |
| （新增）tutorial.ts | `core/tutorial.ts`：3 卡轻量新手引导（投放合成/编队拖拽/战术技，按 uid 持久化） |
| render.js | **未被 prefab 替代**：落到 `view/Painter.ts`（即时 canvas 绘制原语）+ `view/GameApp.ts`（各场景绘制/事件分发主入口）+ `view/Tex.ts`（即梦 AI 美术图集加载）；即时绘制方案保留，非节点/预制体 |
| input.js / ui.js | 并入 `view/GameApp.ts` 的指针分发与按钮绘制 |
| main.js | 主场景脚本 + `core/flow.ts` 接线 |

## 七、分步重构计划（每步跑测试、单独提交，随时可回滚）

1. **抽数据层**：config.js / story-data.js / state.js，game.js 引用改为 SF.C / SF.STORY / SF.G → 跑 Playwright 全流程测试 → commit。
2. **抽服务层**：fx.js / synergy.js / fleet.js（几乎纯函数，风险最低）→ 测试 → commit。
3. **抽系统层**：forge / board / battle / economy / ads / story 六个，逐个搬 → 每搬一个测一次 → commit。
4. **抽表现与交互层**：render.js（最大块）、input.js、ui.js、main.js，删除 game.js → 全量回归（合成/拖拽/升星/战斗/经济/广告/剧情七条链路）→ commit。
5. **更新构建**：单文件版脚本改为列表拼接；README 更新目录说明 → commit。

每步验收标准一致：Playwright 零报错 + 关键链路行为与重构前完全一致（不改任何玩法数值和手感）。

## 八、风险与对策

加载顺序敏感 → 顺序只写在 index.html 和构建脚本两处，README 注明"新增模块要同时改这两处"。全局命名冲突 → 只允许 `SF` 一个全局，review 时搜 `window.` 即可审计。拆分引入回归 → 每步小提交 + 自动化测试兜底，坏了回滚单个 commit。
